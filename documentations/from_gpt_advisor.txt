# 1. Setup akun yang perlu dibuat

## A. Vercel

Buat project Next.js di Vercel.

Vercel akan host:

```txt id="la2lgm"
/
/create
/preview
/success
/api/orders
/api/upload-url
/api/orders/[orderId]
/api/webhooks/doku
```

Next.js Route Handlers memang dipakai untuk bikin custom request handlers di `app/api/.../route.ts`, support method seperti GET/POST, dan hanya tersedia di `app` directory. ([Next.js][2])

Jadi yes: **frontend dan API routes bisa satu project di Vercel.**

---

## B. Cloudflare R2

Buat R2 bucket, misalnya:

```txt id="brkq2x"
love-cards-prod
love-cards-staging
```

R2 akan menyimpan:

```txt id="7olhj5"
pending draft
uploaded photos
uploaded audio
generated static HTML
config.json
QR
published cards
```

R2 mendukung S3-compatible API, jadi di Next.js kamu bisa pakai AWS S3 SDK dengan endpoint R2. ([Cloudflare Docs][3])

---

## C. Public domain untuk R2

Idealnya kamu punya 2 domain/subdomain:

```txt id="wrieed"
app.yourdomain.com
→ Vercel app

pub.yourdomain.com
→ R2 public bucket
```

Contoh:

```txt id="f7l9ug"
https://app.lovething.id/create
https://pub.lovething.id/cards/c5aa9556/index.html
```

Cloudflare R2 public buckets bisa dihubungkan ke custom domain dari dashboard R2; kalau domain sudah dikelola Cloudflare, kamu bisa langsung connect bucket ke custom domain. ([Cloudflare Docs][4])

Untuk V0, public URL boleh begini:

```txt id="m3ykvn"
https://pub.yourdomain.com/cards/{cardId}/index.html
```

Nanti kalau mau URL cantik:

```txt id="vndz7w"
https://yourdomain.com/card/{cardId}
```

baru tambahkan routing layer.

# 2. Setup webhook DOKU

Kamu perlu webhook URL:

```txt id="vx5mvj"
https://app.yourdomain.com/api/webhooks/doku
```

DOKU webhook/payment notification dipakai supaya DOKU bisa mengirim update status pembayaran real-time ke sistem merchant, jadi kamu tidak perlu polling terus-menerus. ([DOKU Docs][6])

Di backend, webhook wajib:

```txt id="z5n3bl"
1. Verify signature dari DOKU
2. Check order id
3. Check amount
4. Check payment status
5. Publish card kalau paid
6. Return 2xx
```

DOKU best practice juga menekankan HTTP Notification harus diverifikasi dengan signature di request header. ([DOKU API][7])

Tambahkan juga fallback endpoint:

```txt id="t9uhwn"
GET /api/orders/[orderId]
```

Kalau webhook telat, success page bisa polling. DOKU juga punya Check Status API untuk cek status transaksi, dengan catatan hit setelah 60 detik setelah payment completion. ([DOKU API][8])

# 3. Apakah perlu “publisher” lagi?

**Perlu sebagai module, bukan service terpisah.**

Jangan bayangin ada server publisher sendiri. Cukup bikin function:

```txt id="22dymq"
publishCard(orderId)
```

Lokasinya:

```txt id="mxq4n0"
src/lib/publisher/publish-card.ts
```

Dipanggil dari webhook:

```txt id="kk53io"
/api/webhooks/doku
↓
verify payment
↓
publishCard(orderId)
```

Publisher melakukan:

```txt id="hc6tr5"
1. Load pending draft dari R2
2. Load assets dari R2
3. Generate config.json
4. Generate index.html
5. Generate qr.svg / qr.png
6. Copy/move assets ke folder cards/
7. Write status published
```

Jadi komponen “publisher” tetap ada, tapi cuma kode internal.

# 5. Endpoint yang perlu dibuat

## 1. `POST /api/upload-url`

Untuk upload foto/audio langsung ke R2.

Kenapa perlu? Supaya file besar tidak lewat Vercel.

Flow:

```txt id="cpvqvh"
Frontend minta signed upload URL
↓
API generate pre-signed R2 URL
↓
Browser upload file langsung ke R2
↓
Frontend simpan key file
```

Request:

```json id="zy2p72"
{
  "fileName": "photo-1.jpg",
  "contentType": "image/webp",
  "fileSize": 850000
}
```

Response:

```json id="id5vpv"
{
  "uploadUrl": "https://...",
  "key": "pending_uploads/tmp_abc/photo-1.webp"
}
```

Untuk V0 super cepat, kamu bisa skip ini dan upload via `/api/orders`, tapi gue sarankan langsung pakai signed upload URL.

---

## 2. `POST /api/orders`

Tugas:

```txt id="vn2p1f"
Generate orderId
Generate cardId
Validate draft
Save draft.json ke R2
Create DOKU Checkout
Save order.json ke R2
Return paymentUrl
```

Response:

```json id="qp0k0w"
{
  "orderId": "order_abc123",
  "cardId": "card_c5aa9556",
  "amount": 5000,
  "paymentUrl": "https://checkout.doku.com/..."
}
```

---

## 3. `POST /api/webhooks/doku`

Tugas:

```txt id="dzh8mq"
Receive webhook
Save raw webhook event ke R2
Verify DOKU signature
Check status paid/success
Check amount = 5000
Check order exists
Check not already published
Call publishCard(orderId)
Return 200
```

Webhook harus idempotent.

Artinya kalau DOKU kirim event yang sama beberapa kali:

```txt id="9qtkqm"
kalau sudah published → return 200
jangan generate ulang terus
```

---

Response kalau masih pending:

Response kalau published:

# 7. Renderer static

Renderer final card bisa 2 pendekatan.

## Opsi A — inline renderer per card

`index.html` berisi semua CSS/JS basic.

Pros:

```txt id="xpeagt"
simple
tidak perlu asset renderer global
semua self-contained
```

Cons:

```txt id="mjg6t7"
index.html lebih besar
kalau update renderer, card lama tidak ikut update
```

## Opsi B — shared renderer global

`index.html` kecil:

```html id="fa0l19"
<div id="app"></div>
<script>
  window.__CARD_CONFIG_URL__ = "./config.json";
</script>
<script src="https://pub.yourdomain.com/renderer/v1/app.js"></script>
```

Pros:

```txt id="5wi15r"
lebih rapi
bisa reusable
index.html kecil
```

Cons:

```txt id="u516kc"
perlu build renderer terpisah
kalau renderer v1 rusak, semua card v1 kena
```

Rekomendasi gue:

```txt id="x4xi3c"
V0: inline/simple renderer dulu
V1: shared renderer /renderer/v1/app.js
```

Biar cepat launch.


# 13. Data schema V0

Gunakan Zod dari awal supaya AI/vibe coding tidak ngaco.

```ts id="j9qrt6"
import { z } from "zod"

export const CardDraftSchema = z.object({
  template: z.literal("flower_secret_letter"),
  fromName: z.string().min(1).max(40),
  toName: z.string().min(1).max(40),
  secretCode: z.string().min(1).max(12),
  letterTitle: z.string().max(80).optional(),
  letterBody: z.string().min(1).max(3000),
  photos: z.array(
    z.object({
      key: z.string(),
      caption: z.string().max(120).optional()
    })
  ).max(5),
  voiceNote: z.object({
    key: z.string(),
    durationSeconds: z.number().max(60).optional()
  }).optional()
})

export type CardDraft = z.infer<typeof CardDraftSchema>
```

Order status:

```ts id="uyhiyv"
export const OrderStatusSchema = z.enum([
  "draft_created",
  "pending_payment",
  "paid",
  "publishing",
  "published",
  "failed"
])
```

# 14. State machine order

Ini penting biar webhook tidak kacau.

```txt id="wiu48o"
draft_created
↓
pending_payment
↓
paid
↓
publishing
↓
published
```

Failure states:

```txt id="eixbn0"
payment_failed
payment_expired
publish_failed
```

Webhook logic:

```txt id="zriywq"
if unpaid → save event, return 200
if already published → return 200
if paid → publish
if publish success → status published
if publish fail → status publish_failed
```
